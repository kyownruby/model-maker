# キャラクターモデル制作 & Claude Code操作システム 設計書

> **この設計書について**
> この文書は、Claude Code（Fable 5）に渡して実装を進めてもらうためのハンドオフ設計書です。
> 「1枚絵／三面図 → Live2D・3Dモデル化 → パラメータ操作 → Claude Code（MCP）から遠隔操作」までの一連のシステムを構築することを目的とします。
> Blenderの Claude 連携（blender-mcp）と同じ思想で、**モデルビューアを「操作対象」、MCPサーバーを「司令塔」**として設計しています。

---

## 0. ゴール（このシステムで実現したいこと）

1. キャラクターの**1枚絵、または三面図（正面・側面・背面）**を入力にする
2. そこから **Live2Dモデル（2D）** と **3Dモデル（VRM等）** を作成する
3. 表情差分イラストがあれば、それも表情パラメータに組み込む
4. 完成後、**表情・ポーズ・目線・口・体の向きなどのパラメータを触って動かせる**
5. **Claude Code と MCP 連携**し、チャットから「笑って」「右向いて」等の指示でモデルを操作できる
6. 最終的には **配信・録画（OBS）に映す**ところまで対応する

---

## 1. 全体アーキテクチャ

役割を3つに分けて考えます。**ここが設計の肝です。**

```
┌────────────────────────────────────────────────────────┐
│  [1] Claude Code (Fable 5)  ← ご主人がチャットで指示     │
│      「ミアを笑顔にして、右を向かせて」                 │
└───────────────┬────────────────────────────────────────┘
                │ MCP (stdio / SSE)
                ▼
┌────────────────────────────────────────────────────────┐
│  [2] MCPサーバー（司令塔 / Windowsローカルで常駐）      │
│      - Claude Codeからの指示を受け取る                  │
│      - パラメータ操作コマンドに変換                     │
│      - ビューアへ WebSocket で送信                      │
└───────────────┬────────────────────────────────────────┘
                │ WebSocket (localhost)
                ▼
┌────────────────────────────────────────────────────────┐
│  [3] モデルビューア（操作対象 / ブラウザ or Electron）  │
│      - Live2D / VRM を表示                              │
│      - 受け取ったパラメータでモデルを動かす             │
│      - 透過背景 → OBSにブラウザソースとして取り込み     │
└────────────────────────────────────────────────────────┘
```

### なぜこの3分割なのか
- **MCPサーバーは常時動くプログラム**なので、GitHub Pages（静的サイト）には置けない → **Windowsローカルで動かす**
- **ビューアは表示するだけ**なので、ローカルでもWeb（GitHub Pages）でも置ける → **共有・公開用にGitHub Pagesを使える**
- この分離により「頭脳（MCP）」と「顔（ビューア）」を独立して開発・差し替えできる

## 2. 技術スタック

### 2-1. モデルビューア（表示側）
| 項目 | 採用技術 | 備考 |
|------|----------|------|
| Live2D表示 | **Live2D Cubism SDK for Web (pixi-live2d-display)** | ブラウザでLive2Dを描画 |
| 3D(VRM)表示 | **three.js + @pixiv/three-vrm** | VRMの標準的な表示ライブラリ |
| 実行形態 | ブラウザ（推奨）／Electron（OBS常駐したいなら） | まずはブラウザでOK |
| 通信 | WebSocket クライアント | MCPサーバーからの指示を受信 |

### 2-2. MCPサーバー（司令塔）
| 項目 | 採用技術 | 備考 |
|------|----------|------|
| 言語 | **Node.js (TypeScript)** | ビューアと同じJS系で統一、WebSocketが扱いやすい |
| MCP | **@modelcontextprotocol/sdk** | 公式MCP SDK |
| 通信 | stdio（Claude Codeとの接続）+ WebSocketサーバー（ビューアとの接続） | |

### 2-3. モデル制作の自動化（工程支援）
| 工程 | ツール | 自動化できる範囲 |
|------|--------|------------------|
| 画像レイヤー分割（Live2D用） | rembg + Pillow / Segment Anything | 前髪・後髪・目・眉・口・体などをパーツ分割 |
| Live2Dモデリング | **Live2D Cubism Editor**（GUI手作業が中心） | ⚠️ 完全自動化は困難。半自動が現実的 |
| 3D化 | 画像→3D（TripoSR / Wonder3D 等）or VRoid Studio | VRoidで手作業のほうが品質安定 |

> **⚠️ 重要な現実ライン**：Live2Dの「リギング（動かすための骨組み設定）」と3Dモデルの造形は、
> 現状フルオート化が難しい領域です。この設計書では**「AIが下処理（パーツ分割・素体生成）を担い、
> 仕上げは既存GUIツール」**という現実的なハイブリッド方針を採ります。ここはFableに
> 「どこまで自動化可能か」を調査・提案させるのが良いです。

## 3. 実装フェーズ（Fableへの作業指示）

段階的に作ると失敗しにくいので、フェーズ分けします。

### Phase 1: ビューア基盤（まず動くものを）
- [ ] three.js + three-vrm で **VRMを1体表示**するWebページをつくるVRM
- [ ] VRMの表情（BlendShape）とボーン（首・目線）をスライダーUIで動かせるようにする
- [ ] pixi-live2d-display で **Live2Dサンプルモデルを表示**、パラメータをスライダーで動かせるようにする
- **完了条件**：ブラウザ上で手動スライダーからモデルの表情・向きが変わる

### Phase 2: MCPサーバー & 遠隔操作
- [ ] Node.js + MCP SDK で MCPサーバーを作成
- [ ] ビューアに WebSocket クライアントを追加、MCPサーバーと接続
- [ ] MCPツールを実装（下記 4章のツール仕様）
- [ ] Claude Code に MCPサーバーを登録し、チャットから操作できることを確認
- **完了条件**：Claude Codeで「笑顔にして」と言うとビューアのモデルが笑う

### Phase 3: モデル制作パイプライン
- [ ] 1枚絵／三面図を入力に、Live2D用のパーツ分割を半自動化するスクリプト
- [ ] 表情差分イラストを取り込み、表情モーフに割り当てる仕組み
- [ ] 3D化ルート（VRoid手動 or 画像→3D）の手順を確立
- [ ] エクスポート機能を実装（Live2D=zip書き出し、3D=VRM/GLB書き出し、MMDは変換ルート調査）
- **完了条件**：添付イラストから最低限動くモデルが1体でき、VRM または Live2D一式でダウンロードできる

### Phase 4: OBS配信対応
- [ ] ビューアの背景を透過にする
- [ ] OBSのブラウザソース（またはウィンドウキャプチャ）で取り込めることを確認
- [ ] （任意）音声連動で口パクさせる（既存のVOICEPEAK音声パイプラインと連携可能）
- **完了条件**：OBS画面にモデルが透過で乗り、Claude Codeから操作できる

## 4. MCP ツール仕様（Claude Codeから呼べる操作）

MCPサーバーが提供するツール。Claude Codeはこれらを呼んでモデルを操作します。

| ツール名 | 引数 | 説明 |
|----------|------|------|
| `set_expression` | `name`（例: smile / angry / sad / surprised）, `weight`(0.0〜1.0) | 表情を設定 |
| `set_pose` | `name`（例: wave / think / point）| プリセットポーズを再生 |
| `look_at` | `x`(-1.0〜1.0), `y`(-1.0〜1.0) | 目線・顔の向きを設定 |
| `set_body_angle` | `x`, `y`, `z` | 体の傾き・向き |
| `play_motion` | `name` | モーション（アニメ）を再生 |
| `set_mouth` | `open`(0.0〜1.0) | 口の開き（口パク用） |
| `list_parameters` | なし | 現在のモデルで操作可能なパラメータ一覧を返す |
| `load_model` | `path` | 別モデルを読み込む |

### ツール設計の指針
- **Fableへ**：まずは `list_parameters` を実装し、モデルが持つパラメータをClaude側が把握できるようにすること。
  これがあると「このモデルには何ができるか」をClaude Codeが自分で調べて操作できる（Blender-MCPと同じ発想）。
- 引数は正規化された値（0.0〜1.0 / -1.0〜1.0）で統一し、Live2D/VRMの内部差をMCPサーバー側で吸収する。

## 4-2. モデルのエクスポート / ダウンロード機能

作成したモデルを、他アプリ（VTube Studio / VRChat / MMD 等）で使えるように書き出す機能。
ビューアに「ダウンロード」ボタンを設け、MCP経由でも `export_model` ツールで呼べるようにする。

### Live2D のエクスポート形式
| ファイル | 内容 | 用途 |
|----------|------|------|
| `.moc3` | モデル実データ | VTube Studio等が読み込むメインファイル（動作専用・編集不可） |
| `.model3.json` | テクスチャ等の設定情報 | モデル一式の設定ファイル |
| `.motion3.json` | モーション／アニメデータ | 動きのデータ |
| `.psd` | パーツ分解された素材 | 制作元データ（Cubism Editorへ再インポート可・編集用） |

> Live2Dは上記を**まとめてzip**で書き出すのが実用的（VTube Studioはフォルダ単位で読む）。

### 3D のエクスポート形式
| 形式 | 対応度 | 用途 | 備考 |
|------|--------|------|------|
| **VRM** | ◎ 本命 | VTube Studio / VRChat / VSeeFace | 配信・共有の定番。まずこれを確実に |
| **glTF/GLB** | ○ | 汎用3D、Web表示 | three.jsからそのまま書き出せる |
| **FBX** | △ | Unity/Blender連携 | 中間フォーマットとして |
| **MMD (.pmx/.pmd)** | △ 変換ルート | MMD | ⚠️ VRMから直接は出せない。**VRM→PMX変換ツール経由**が必要 |

> **⚠️ MMDについて（Fableへ）**：VRMからMMD(.pmx)への変換は、標準ライブラリでは一発では出せません。
> `VRM2PMXConverter` 等の外部変換ツール、またはBlender（mmd_tools + VRMアドオン）経由になります。
> 「ボーン構造・物理・表情モーフの互換」に手作業補正が入る前提で、**最新の変換手法をWeb調査してから**
> 実装可否と手順を提案してください。まずは **VRM / GLB を確実に**、MMDは「変換ルートとして用意」の位置づけ。

### 追加するMCPツール
| ツール名 | 引数 | 説明 |
|----------|------|------|
| `export_model` | `format`（vrm / glb / fbx / live2d_zip / pmx）, `path` | 指定形式でモデルを書き出し |
| `list_export_formats` | なし | そのモデルで書き出せる形式一覧を返す |

## 5. 入力データの仕様（今回の素材）

- **三面図**：正面・側面・背面（添付あり）。Live2Dはほぼ正面、3Dは三面図が有効。
- **1枚絵**：正面立ち絵（添付あり）。
- **表情差分**：あれば表情モーフの目標形状として使用。**Fableへ**：表情差分の命名規則（例: `face_smile.png`）を提案・統一させること。
- **キャラ設定**：Craftのキャラプロンプト／Personaドキュメントと整合を取る（イメージカラー・髪型の特徴など）。

## 6. 実行環境メモ（Windowsローカル前提）

- **必須**：Node.js（LTS）、モダンブラウザ（Chrome/Edge）
- **MCPサーバー起動**：`node server.js`（stdio接続でClaude Codeに登録）
- **Claude Code登録例**（`.mcp.json` イメージ）：
```json
{
  "mcpServers": {
    "character-model": {
      "command": "node",
      "args": ["C:/path/to/mcp-server/dist/server.js"]
    }
  }
}
```
- **ビューア**：ローカルは `http://localhost:xxxx`、共有版は GitHub Pages に静的ホスト（※MCP接続はローカル時のみ）

## 7. リスク & 事前に決めておくこと（Fableへの確認事項）

Fableには、実装前に以下を調査・確認してから進めてもらうと安全です。

1. **Live2Dのライセンス**：Cubism SDKは規模により商用ライセンスが必要な場合あり。用途（個人／収益化）に応じて確認。
2. **モデル制作の自動化限界**：Live2Dリギングと3D造形はどこまで自動化できるか、最新ツールを**必ず事前Web調査**して現実解を提案する。
3. **VRM vs 独自3D**：配信で使うなら VRM が無難（VTube Studio等の資産も使える）。要判断。
4. **口パク音声連携**：既存のVOICEPEAK/VB-CABLE音声パイプラインと繋ぐか。
5. **既存資産との整合**：キャラのイメージカラー・髪型設定はCraftのドキュメントを正とする。
